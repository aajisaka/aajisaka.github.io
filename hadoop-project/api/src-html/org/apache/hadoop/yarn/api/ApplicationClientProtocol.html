092/**
093 * <p>The protocol between clients and the <code>ResourceManager</code>
094 * to submit/abort jobs and to get information on applications, cluster metrics,
095 * nodes, queues and ACLs.</p>

101   * <p>The interface used by clients to obtain a new {@link ApplicationId} for 
102   * submitting new applications.</p>
103   * 
104   * <p>The <code>ResourceManager</code> responds with a new, monotonically
105   * increasing, {@link ApplicationId} which is used by the client to submit
106   * a new application.</p>
107   *
108   * <p>The <code>ResourceManager</code> also responds with details such 
109   * as maximum resource capabilities in the cluster as specified in
110   * {@link GetNewApplicationResponse}.</p>
111   *
112   * @param request request to get a new <code>ApplicationId</code>
113   * @return response containing the new <code>ApplicationId</code> to be used
114   * to submit an application
115   * @throws YarnException
116   * @throws IOException
117   * @see #submitApplication(SubmitApplicationRequest)
118   */
119  @Public
120  @Stable
121  @Idempotent
122  public GetNewApplicationResponse getNewApplication(
123      GetNewApplicationRequest request)
124  throws YarnException, IOException;

126  /**
127   * <p>The interface used by clients to submit a new application to the
128   * <code>ResourceManager.</code></p>
129   * 
130   * <p>The client is required to provide details such as queue, 
131   * {@link Resource} required to run the <code>ApplicationMaster</code>, 
132   * the equivalent of {@link ContainerLaunchContext} for launching
133   * the <code>ApplicationMaster</code> etc. via the 
134   * {@link SubmitApplicationRequest}.</p>
135   * 
136   * <p>Currently the <code>ResourceManager</code> sends an immediate (empty) 
137   * {@link SubmitApplicationResponse} on accepting the submission and throws 
138   * an exception if it rejects the submission. However, this call needs to be
139   * followed by {@link #getApplicationReport(GetApplicationReportRequest)}
140   * to make sure that the application gets properly submitted - obtaining a
141   * {@link SubmitApplicationResponse} from ResourceManager doesn't guarantee
142   * that RM 'remembers' this application beyond failover or restart. If RM
143   * failover or RM restart happens before ResourceManager saves the
144   * application's state successfully, the subsequent
145   * {@link #getApplicationReport(GetApplicationReportRequest)} will throw
146   * a {@link ApplicationNotFoundException}. The Clients need to re-submit
147   * the application with the same {@link ApplicationSubmissionContext} when
148   * it encounters the {@link ApplicationNotFoundException} on the
149   * {@link #getApplicationReport(GetApplicationReportRequest)} call.</p>
150   * 
151   * <p>During the submission process, it checks whether the application
152   * already exists. If the application exists, it will simply return
153   * SubmitApplicationResponse</p>
154   *
155   * <p> In secure mode,the <code>ResourceManager</code> verifies access to
156   * queues etc. before accepting the application submission.</p>
157   * 
158   * @param request request to submit a new application
159   * @return (empty) response on accepting the submission
160   * @throws YarnException
161   * @throws IOException
162   * @throws InvalidResourceRequestException
163   *           The exception is thrown when a {@link ResourceRequest} is out of
164   *           the range of the configured lower and upper resource boundaries.
165   * @see #getNewApplication(GetNewApplicationRequest)
166   */
167  @Public
168  @Stable
169  @Idempotent
170  public SubmitApplicationResponse submitApplication(
171      SubmitApplicationRequest request) 
172  throws YarnException, IOException;

560  /**
561   * <p>
562   * The interface used by clients to submit a new reservation to the
563   * {@code ResourceManager}.
564   * </p>
565   * 
566   * <p>
567   * The client packages all details of its request in a
568   * {@link ReservationSubmissionRequest} object. This contains information
569   * about the amount of capacity, temporal constraints, and concurrency needs.
570   * Furthermore, the reservation might be composed of multiple stages, with
571   * ordering dependencies among them.
572   * </p>
573   * 
574   * <p>
575   * In order to respond, a new admission control component in the
576   * {@code ResourceManager} performs an analysis of the resources that have
577   * been committed over the period of time the user is requesting, verify that
578   * the user requests can be fulfilled, and that it respect a sharing policy
579   * (e.g., {@code CapacityOverTimePolicy}). Once it has positively determined
580   * that the ReservationSubmissionRequest is satisfiable the
581   * {@code ResourceManager} answers with a
582   * {@link ReservationSubmissionResponse} that include a non-null
583   * {@link ReservationId}. Upon failure to find a valid allocation the response
584   * is an exception with the reason.
585   * 
586   * On application submission the client can use this {@link ReservationId} to
587   * obtain access to the reserved resources.
588   * </p>
589   * 
590   * <p>
591   * The system guarantees that during the time-range specified by the user, the
592   * reservationID will be corresponding to a valid reservation. The amount of
593   * capacity dedicated to such queue can vary overtime, depending of the
594   * allocation that has been determined. But it is guaranteed to satisfy all
595   * the constraint expressed by the user in the
596   * {@link ReservationSubmissionRequest}.
597   * </p>
598   * 
599   * @param request the request to submit a new Reservation
600   * @return response the {@link ReservationId} on accepting the submission
601   * @throws YarnException if the request is invalid or reservation cannot be
602   *           created successfully
603   * @throws IOException
604   * 
605   */
606  @Public
607  @Unstable
608  public ReservationSubmissionResponse submitReservation(
609      ReservationSubmissionRequest request) throws YarnException, IOException;
610
611  /**
612   * <p>
613   * The interface used by clients to update an existing Reservation. This is
614   * referred to as a re-negotiation process, in which a user that has
615   * previously submitted a Reservation.
616   * </p>
617   * 
618   * <p>
619   * The allocation is attempted by virtually substituting all previous
620   * allocations related to this Reservation with new ones, that satisfy the new
621   * {@link ReservationUpdateRequest}. Upon success the previous allocation is
622   * substituted by the new one, and on failure (i.e., if the system cannot find
623   * a valid allocation for the updated request), the previous allocation
624   * remains valid.
625   * 
626   * The {@link ReservationId} is not changed, and applications currently
627   * running within this reservation will automatically receive the resources
628   * based on the new allocation.
629   * </p>
630   * 
631   * @param request to update an existing Reservation (the ReservationRequest
632   *          should refer to an existing valid {@link ReservationId})
633   * @return response empty on successfully updating the existing reservation
634   * @throws YarnException if the request is invalid or reservation cannot be
635   *           updated successfully
636   * @throws IOException
637   * 
638   */
639  @Public
640  @Unstable
641  public ReservationUpdateResponse updateReservation(
642      ReservationUpdateRequest request) throws YarnException, IOException;
